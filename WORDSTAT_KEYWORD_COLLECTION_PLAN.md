# План полуавтоматического сбора ключей из Яндекс Wordstat

## Цель

Сделать рабочий процесс, в котором Codex помогает собирать ключевые фразы по теме через браузерную автоматизацию, а пользователь вручную проходит авторизацию, капчу и любые действия, которые требуют подтверждения человеком.

Основная идея: не пытаться напрямую подключаться к личному Chrome-профилю пользователя, а запускать отдельный управляемый браузерный профиль Playwright. Пользователь один раз входит в Яндекс в этом окне, после чего сессия сохраняется локально, а сборщик выполняет рутинные действия: открывает Wordstat, вводит seed-запросы, переключает регион, читает таблицы, сохраняет CSV/JSON и делает первичную очистку.

## Принципы

- Не обходить капчу, антибот-защиту и лимиты сервиса.
- Использовать отдельный браузерный профиль, чтобы не трогать основной Chrome пользователя.
- Делать сбор с человеческими задержками и возможностью паузы.
- Сохранять сырые данные отдельно от обработанных.
- Делать сбор воспроизводимым: все входные запросы, регион, дата, режим и версия скрипта должны попадать в артефакты.
- Не полагаться на хрупкие CSS-селекторы без проверки: сначала нужен режим записи/диагностики, затем стабильные локаторы.

## Что нужно от пользователя

Перед запуском пользователь передает:

- тему или список стартовых seed-запросов;
- регион или несколько регионов;
- язык и географию бизнеса;
- стоп-темы и очевидные минус-слова;
- глубину сбора: только стартовые фразы, расширение по похожим фразам или несколько итераций;
- желаемый формат результата: CSV, JSON, Markdown-отчет, структура SEO-страниц или структура рекламных групп.

## Рекомендуемая структура файлов

```text
scripts/
  wordstat/
    collect-wordstat.mjs
    parse-wordstat.mjs
    cluster-keywords.mjs
    config.example.json
data/
  wordstat/
    seeds.txt
    raw/
    processed/
    reports/
.auth/
  wordstat-playwright-profile/
```

`.auth/` лучше добавить в `.gitignore`, потому что там будет храниться локальная браузерная сессия.

## Пошаговый план выполнения

### Шаг 1. Подготовить входные данные

Создать файл `data/wordstat/seeds.txt`.

Пример:

```text
разработка сайтов
создание сайта под ключ
seo продвижение сайта
лендинг под ключ
```

Создать конфиг `scripts/wordstat/config.json`.

Пример:

```json
{
  "regions": ["Москва", "Санкт-Петербург"],
  "maxDepth": 1,
  "maxPagesPerQuery": 3,
  "delayMs": {
    "min": 2500,
    "max": 7000
  },
  "pauseOnCaptcha": true,
  "outputDir": "data/wordstat"
}
```

### Шаг 2. Запустить браузер с отдельным профилем

Скрипт запускает Chromium или Chrome через Playwright в persistent context:

```bash
node scripts/wordstat/collect-wordstat.mjs --login
```

Логика режима `--login`:

1. Открыть Wordstat.
2. Дать пользователю вручную войти в Яндекс.
3. Дождаться подтверждения в терминале.
4. Сохранить состояние профиля в `.auth/wordstat-playwright-profile/`.

После этого повторная авторизация обычно не нужна, пока Яндекс не сбросит сессию.

### Шаг 3. Проверить селекторы и сценарий

Перед массовым сбором нужен тестовый прогон на 1 seed-запросе.

Скрипт должен проверить:

- поле ввода запроса найдено;
- запрос вводится корректно;
- регион выбран или уже установлен;
- результаты загрузились;
- строки с ключами и частотностью считываются;
- обнаруживается состояние капчи или ошибки;
- экспорт создает корректный файл.

Команда:

```bash
node scripts/wordstat/collect-wordstat.mjs --dry-run --seed "разработка сайтов" --region "Москва"
```

Если верстка Wordstat изменилась, скрипт должен не падать молча, а сохранить скриншот и HTML-фрагмент в `data/wordstat/debug/`.

### Шаг 4. Собрать сырые данные

Основной запуск:

```bash
node scripts/wordstat/collect-wordstat.mjs --config scripts/wordstat/config.json --seeds data/wordstat/seeds.txt
```

Для каждого seed-запроса и региона сборщик:

1. Открывает Wordstat.
2. Устанавливает регион.
3. Вводит запрос.
4. Дожидается загрузки результатов.
5. Считывает фразы, частотность и тип блока.
6. Переходит по страницам, если пагинация доступна.
7. Сохраняет сырой результат.
8. Ставит случайную задержку.
9. Продолжает следующий запрос.

Сырые файлы лучше сохранять отдельно по дате:

```text
data/wordstat/raw/2026-07-07/moskva__razrabotka-saitov.json
data/wordstat/raw/2026-07-07/spb__razrabotka-saitov.json
```

### Шаг 5. Обработка и очистка

После сбора запускается обработчик:

```bash
node scripts/wordstat/parse-wordstat.mjs --input data/wordstat/raw/2026-07-07 --output data/wordstat/processed/keywords.csv
```

Он делает:

- нормализацию регистра;
- удаление дублей;
- удаление пустых и битых строк;
- объединение частотностей по регионам;
- пометку исходного seed-запроса;
- пометку региона;
- удаление фраз по минус-словам;
- выделение базовых токенов;
- первичную классификацию интента.

Рекомендуемые колонки CSV:

```text
keyword,frequency,region,source_seed,source_block,collected_at,intent,cluster,notes
```

### Шаг 6. Расширение семантики

Если нужен сбор в глубину, обработчик выбирает новые seed-запросы из найденных фраз.

Правила отбора:

- брать фразы с частотностью выше заданного порога;
- не брать слишком длинные фразы на первой итерации;
- исключать брендовые, мусорные и нерелевантные запросы;
- не добавлять уже обработанные запросы;
- ограничивать количество новых seed-запросов на одну итерацию.

Пример логики:

```text
queue = стартовые seed-запросы
visited = пусто

пока queue не пуста и depth <= maxDepth:
  взять следующий запрос
  если запрос уже в visited:
    пропустить
  собрать Wordstat
  сохранить raw
  извлечь кандидатов для расширения
  добавить кандидатов в queue
  добавить запрос в visited
```

### Шаг 7. Кластеризация

Кластеризация может быть простой или расширенной.

Простая логика:

- группировать по общим леммам;
- выделять коммерческие маркеры: `купить`, `заказать`, `цена`, `стоимость`, `под ключ`;
- выделять информационные маркеры: `как`, `что такое`, `пример`, `инструкция`, `сколько`;
- отделять геозапросы;
- отделять брендовые запросы;
- отделять нерелевантные запросы.

Расширенная логика:

- считать похожесть фраз по токенам;
- кластеризовать по SERP, если отдельно подключить сбор поисковой выдачи;
- использовать эмбеддинги для смысловой группировки;
- делать ручную валидацию спорных кластеров.

### Шаг 8. Финальный отчет

Результат сохраняется в `data/wordstat/reports/`.

Форматы:

- `keywords.csv` - все очищенные ключи;
- `clusters.csv` - кластеры и интенты;
- `minus-words.txt` - предложенные минус-слова;
- `seo-pages.md` - идеи SEO-страниц;
- `ads-groups.csv` - черновик групп для рекламы.

## Логика автоматизации

Сборщик лучше строить как конечный автомат:

```text
INIT
  -> OPEN_WORDSTAT
  -> CHECK_AUTH
  -> SET_REGION
  -> ENTER_QUERY
  -> WAIT_RESULTS
  -> DETECT_CAPTCHA_OR_BLOCK
  -> PARSE_RESULTS
  -> NEXT_PAGE_OR_FINISH_QUERY
  -> SAVE_RESULT
  -> DELAY
  -> NEXT_QUERY
  -> DONE
```

Состояние капчи:

```text
если найдена капча или экран проверки:
  сохранить скриншот
  вывести пользователю сообщение
  поставить процесс на паузу
  ждать Enter в терминале
  повторить проверку страницы
```

Состояние ошибки:

```text
если результаты не загрузились:
  повторить 1-2 раза
  увеличить задержку
  сохранить debug-артефакты
  перейти к следующему запросу
  записать ошибку в лог
```

## Почему не использовать основной Chrome напрямую

Технически можно запускать Chrome с существующим профилем, но это рискованнее:

- профиль может быть заблокирован другим открытым Chrome;
- автоматизация может повредить пользовательские настройки;
- в профиле есть лишние cookies и расширения;
- сложнее воспроизводить проблему;
- выше риск случайно задеть личные вкладки.

Поэтому безопаснее отдельный persistent-профиль Playwright. Он выглядит как обычное браузерное окно, но изолирован от основного браузера.

## Вариант через MCP

Если позже появится MCP-сервер для браузера или Wordstat, логика почти не меняется. MCP просто становится транспортом команд:

```text
open_page(url)
wait_for_login()
set_region(region)
submit_query(seed)
extract_results()
save_artifact()
```

Преимущество MCP: Codex сможет вызывать готовые инструменты вместо отдельного Node.js-скрипта. Недостаток: все равно нужна авторизация пользователя и ручная обработка капчи.

## Минимальный MVP

Первую версию стоит сделать максимально простой:

1. Запуск отдельного браузерного профиля.
2. Ручная авторизация.
3. Сбор 1 региона и списка seed-запросов.
4. Сохранение raw JSON.
5. Экспорт очищенного CSV.
6. Пауза при капче.
7. Debug-скриншоты при ошибках.

После MVP можно добавить:

- несколько регионов;
- расширение seed-запросов;
- кластеризацию;
- минус-слова;
- HTML/Markdown-отчет;
- сравнение частотностей по регионам;
- ручную разметку релевантности.

## Критерии готовности

Процесс можно считать готовым, если:

- пользователь может один раз авторизоваться в отдельном окне;
- сборщик проходит тестовый seed-запрос;
- данные сохраняются в raw JSON;
- создается CSV с ключами и частотностью;
- при капче скрипт останавливается и продолжает после ручного решения;
- повторный запуск не собирает уже обработанные запросы без необходимости;
- все ошибки попадают в лог и debug-папку.
